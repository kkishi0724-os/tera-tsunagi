# 方針転換 v3：離脱を減らす「簡素化・自由入力・直接やりとり」（2026-07-15）

`DESIGN.md` / `DESIGN_V2_PORTAL.md` の**根本方針を更新**。目的は **離脱（登録・掲載・申込の途中離脱）を減らすこと**。
迷わせる設定・必須項目を削り、細部は人と人のやりとりで決める。

## コア原則

1. **掲載は自由入力・必須は最小**
   - お寺は「貸せる場所」と「空いている日時」を**自由記述**で書くだけ。
   - プリセットの場所カード・時間スロット・設備チェック・料金・人数などの**細かい設定は不要**（すべて任意 or 廃止）。
   - 必須は実質 **2項目（場所／空いている日時）** のみ。
2. **申込後は個人間（お寺⇔申込者）の直接やりとり**
   - 日時・人数・条件などの詳細は、申込が来てから**当事者どうしのメッセージで決める**。
   - フォーム側で細かく詰めさせない（＝入口の心理的ハードルを下げる）。
3. **トラブルは運営（自社）が対応する安全網**
   - やりとりは基本 1対1。困りごと・トラブル時に**運営が介入して対応**する（常時三者仲介ではなく、必要時に入る）。
4. **全体を簡単に**
   - どの画面も「必須は最小・あとは任意＋自由記述」。ボタンと導線を減らし、シニアにも分かりやすく。

## 各画面への影響（この方針で作り直す）

| 画面 | Before（重い） | After（この方針） |
|---|---|---|
| 掲載フォーム | 4ステップ（場所/日時/条件・設備/確認）＋プリセット多数 | **1画面・自由入力（場所＋空き日時＋ひとこと任意）**。必須2つ。→ `web/app/temple/new` |
| 企画者申込 / ワーカー応募 | タイトル・ジャンル・希望日時選択・人数・内容…必須多め | 必須を**短いメッセージ中心**に削減。日時は「希望を自由記述」。細部は申込後に直接やりとり |
| 相談スレッド | 寺・企画者・運営の三者が常時同席 | **基本 1対1（お寺⇔申込者）**＋運営はトラブル時に介入（通報/相談で呼べる） |

## データ設計への影響（DESIGN.md / DESIGN_V2 の該当を緩める）

- `spaces` … 厳密な `place_type/capacity/equipment/conditions/base_price` を**必須にしない**。
  代わりに **`space_name`（自由入力）** と **`availability_note`（空き日時の自由記述）** を主とする。
  構造化された `availabilities`（date/start/end）は「任意の補助」に格下げ（無くても掲載可）。
- `applications` / `gig_applications` … `desired_datetime` を**自由テキスト**に。必須は `message` 程度。
- `messages` … 既定は当事者2者（寺⇔ユーザー）。`operator_joined`(bool) 等で運営の介入を表す。
