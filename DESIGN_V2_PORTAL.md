# 設計書 v2：統合ポータル（スペース貸出 × お寺特化スキマバイト）

`DESIGN.md`（v1・スペース貸出のみ）を拡張。お寺の「空きスペース」と「人手不足」を
一つのポータルで解決する。**一般ユーザーは入口で2分岐、お寺側は1つの統合ダッシュボード**。

## 1. コンセプトと全体構造

- お寺の「空きスペース」と「人手不足」を一挙に解決し、若い世代とお寺を繋ぐポータル。
- **フロント（入口の切り分け）**: 総合トップ（ゲートウェイ）で2択を大きく提示。
  - ①**イベント・企画をしたい方**＝スペース・空き時間レンタル窓口（＝v1のテラつなぎ）
  - ②**お寺でお仕事・お手伝いをしたい方**＝お寺専用の単発スキマバイト窓口（新規）
  - ＋ お寺関係者向けのログイン導線。
- **バック（お寺側は統合）**: ダッシュボードは1つ。「スペース貸出」と「スキマバイト募集」を一元管理。

## 2. 全体マップ（ユーザー別の画面遷移ルール）

```
                      ┌─────────────── ゲートウェイ（総合トップ／画面A）───────────────┐
                      │  ①スペースを借りたい    ②お寺でバイトしたい    お寺関係者ログイン │
                      └────────┬────────────────────┬───────────────────────┬──────────┘
                               │                    │                       │
              ①企画者フロー     │       ②ワーカー(学生)フロー │        お寺フロー │
                               ▼                    ▼                       ▼
                  [スペース検索・一覧]        [お手伝い(バイト)一覧]     [統合ダッシュボード／画面B]
                        ▼                          ▼                    ├ ホーム（お知らせ・予定）
                  [スペース詳細→申込]          [募集詳細→応募]           ├ スペース貸出管理
                        ▼                          ▼                    │   （枠掲載・イベント申込承認）
                  [相談スレッド(三者)]         [やりとり・採用連絡]        └ スキマバイト管理
                        │                          │                        （募集作成・応募者マッチング）
                        └──────────► お寺側の各管理メニューに集約 ◄──────────┘
```

**遷移ルールの要点**
- 一般ユーザーは必ずゲートウェイを経由し、**自分の目的（借りる／働く）を選んでから**各フローへ。
- ①②のフローは独立だが、**お寺側では同じダッシュボードに合流**して一元管理。
- お寺関係者はゲートウェイのログイン導線→統合ダッシュボード。左ナビで3メニューを切替。

## 3. データベース設計（v1拡張）

v1（`DESIGN.md`）の `temples / spaces / availabilities / organizers / applications / messages` を継承し、
**ユーザーを統合**し、**バイト系テーブルを追加**する。

```
users（一般ユーザー統合：企画者にもワーカーにもなれる共通アカウント）
  user_id (PK) / name / email / phone /
  is_organizer(bool) / is_worker(bool) /       -- 入口①②のどちらでも利用可
  worker_profile(学校・学年・自己紹介 等 nullable) / created_at
  ※ v1 の organizers は users に統合（organizer_id → user_id へ）

--- スペース貸出（v1・継承） ---
temples / spaces / availabilities は DESIGN.md のまま
applications（スペース申込）: organizer_id を user_id に変更、他は同じ

--- スキマバイト（新規） ---
gigs（お手伝い募集）
  gig_id (PK) / temple_id (FK) / title / category(清掃|祭り補助|受付|SNS/撮影|法要補助 等) /
  description / work_date / start_time / end_time /
  pay_type(時給|日給|謝礼) / pay_amount / headcount(募集人数) /
  requirements(条件・持ち物) / status(募集中|締切|終了) / created_at

gig_applications（バイト応募）
  gig_app_id (PK) / gig_id (FK) / user_id (FK) / message /
  status(応募|面談調整|採用|見送り|完了) / created_at / updated_at

--- 共通 ---
messages（相談・連絡スレッド：スペース申込／バイト応募の両方に対応）
  message_id (PK) / thread_type(space|gig) /
  application_id (FK, nullable) / gig_app_id (FK, nullable) /
  sender_type(寺|ユーザー|運営) / body / created_at
```

**リレーションの要点**
- `temples` 1—* `spaces`（→ `availabilities`）／ `temples` 1—* `gigs`。お寺が両方の供給者。
- `users` 1—* `applications`（スペース）／ `users` 1—* `gig_applications`（バイト）。同一人物が両方可。
- `spaces` ⇄ `users` は `applications`、`gigs` ⇄ `users` は `gig_applications` で多対多。
- `messages` は `thread_type` で space/gig を切替え、どちらのやりとりも運営が仲介可能。
- お寺ダッシュボードは `temple_id` を軸に、`applications`（スペース）と `gig_applications`（バイト）を
  横断集計して「本日のお知らせ・予定」を作る。

## 4. プロトタイプ（画面A・B）

`prototype/portal.html`（1ファイルでA/Bをトグル切替）。既存デザインシステム（`DESIGN_SYSTEM.md`）準拠。
- **画面A ゲートウェイ**: 2大選択（①スペース＝抹茶／②バイト＝朱）をスプリットで提示（ホバーで動的に拡大）＋お寺関係者ログイン導線。
- **画面B 統合ダッシュボード**: 左ナビ（ホーム／スペース貸出管理／スキマバイト管理）。
  ホームに「本日のお知らせ（新着イベント申込・バイト応募の件数）」と「直近スケジュール（スペース利用予定・本日来る学生）」。
